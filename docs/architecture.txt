KEEL — TECHNICAL ARCHITECTURE
Last updated: 2026-05-06 Session 5

═══════════════════════════════════════════════
OVERVIEW
═══════════════════════════════════════════════

Keel is a Gmail-connected inbox intelligence layer. It scans a user's Gmail,
classifies threads by category using AI, and surfaces actionable signals
(payments due, events, awaiting replies etc.) in a structured dashboard.

Two separate Next.js apps:
  keel       — user-facing app  → www.jaison.app / keel-icl6.vercel.app
  keel-admin — internal admin   → keel-admin.vercel.app

SCAN ARCHITECTURE
═══════════════════════════════════════════════

Two paths depending on scan type:

INCREMENTAL / MANUAL (Vercel serverless, ~20s):
  Client → triggerScan('manual'|'auto') → POST /api/gmail/scan
  → getValidAccessToken(): reads token from Firestore, auto-refreshes if expired
  → Gmail API: existence query + recent query (incremental fetch)
  → Skip logic: internalDate vs lastMessageInternalDate (Gmail timestamp, not Keel
    write time — ensures replies always trigger re-classification)
  → manualCategory guard: never overwrite user-assigned categoryId
  → terminal status guard: never overwrite done/archived/paid status
  → S1: classify thread → category + status + aiTitle + aiSummary (with real names,
    signal accuracy rules, NEXT STEP direction in prompt)
  → S2: extract signals (payments, events, awaiting reply etc.)
  → write items to Firestore, write scanRun doc, update meta/usage
  → runCalendarCheck() fire-and-forget

ONBOARDING (Cloud Function, 60min timeout):
  Client → triggerScan('onboarding') → POST gmailscan Cloud Function URL
  → same logic as above but handles full 7–30 day window without timeout risk
  → Cloud Function URL: https://gmailscan-wel4wongwa-ew.a.run.app

REANALYSE SINGLE ITEM (/api/gmail/reanalyse):
  POST { uid, itemId }
  → getValidAccessToken() with auto-refresh
  → fetch thread from Gmail (returns {skipped:true} on 404 — deleted threads)
  → full S1 classification preserving manualCategory if set
  → rewrite signals in Firestore
  → runCalendarCheck() after signal write
  Accessible from ... menu in ItemExpandedPanel and from admin Dev Operations.

DASHBOARD 2.0 MOUNT SCAN:
  On mount, reads lastScanCompletedAt from Firestore account doc.
  Triggers triggerScan('auto') only if > 10 minutes since last scan.
  Shows full-screen blur overlay (z-index 500) with 1.5s minimum display.

Live feed during scan: scan route writes meta/scanFeed every 5 threads.
Client reads meta/scanFeed via onSnapshot for real-time terminal display.

Scan run logging: every scan writes to users/{uid}/scanRuns/{timestamp}
Fields: scanAt, daysBack, threadsFound, threadsProcessed, newItems, updatedItems,
        skipped, inputTokens, outputTokens, aiCostUsd, fbReads, fbWrites,
        fbCostUsd, totalCostUsd, model, provider, job, durationMs

═══════════════════════════════════════════════
CALENDAR CHECK
═══════════════════════════════════════════════

runCalendarCheck(db, uid, accessToken):
  - Reads checkAllCalendars from account doc
  - If false: queries calendars/primary/events only
  - If true: fetches all writable calendars via calendarList API, queries each in
    parallel, pools all events
  - Compares event/rsvp/deadline signals against calendar events in ±1 day window
    using fuzzy title matching (sigDesc vs aiTitle vs calEvent.summary)
  - Writes calendarStatus: 'on_cal' | 'not_on_cal' to signal docs
  - Writes matchedCalendarName when matched on non-Primary calendar
  - Runs after every scan and after every reanalyse

SignalCalendarStatus: 'on_cal' | 'not_on_cal' | 'ignored' | 'pending' | null

Auto-recheck on tab return:
  SignalPill and CalendarBadge listen for visibilitychange when calStatus='pending'.
  When user returns from Google Calendar tab, triggers /api/calendar/check and
  updates local state to 'on_cal' after 1.5s.

Signal calStatus syncs from parent onSnapshot — no page refresh needed.

═══════════════════════════════════════════════
FIRESTORE DATA MODEL
═══════════════════════════════════════════════

/users/{uid}/accounts/account_primary
  { email, displayName, photoURL, daysBack, lastScanCompletedAt,
    categoryIds[], categoryDescriptions{}, emailType,
    checkAllCalendars, accessToken, refreshToken, tokenExpiresAt,
    tokenUpdatedAt, scanCount, lastScanAt }

/users/{uid}/items/{itemId}
  KeelItem: { itemId, threadId, messageId, accountId, senderEmail, senderName,
              subject, receivedAt, categoryId, categoryName,
              manualCategory,        ← true if user assigned (scan won't overwrite)
              status,                ← new|awaiting_action|awaiting_reply|
                                        quietly_logged|done|archived|paid|overdue
              importanceFlag, aiImportanceScore,
              manualPriority, manuallyIgnored,
              mergedThreadIds[], aiTitle, aiSummary, aiDetailedSummary,
              participants[],
              userNote,              ← private note-to-self, never touched by scan
              lastMessageInternalDate, ← Gmail ms timestamp of latest message
              createdAt, updatedAt, resolvedAt,
              snoozedUntil, preSnoozePriority,
              linkedOutboundId, linkedItemId, isRecurring,
              fromTrackedReply, trackedReplyId }

/users/{uid}/signals/{signalId}
  KeelSignal: { signalId, itemId, type, description, detectedDate,
                detectedAmountPence, currency, importanceFlag,
                calendarStatus, matchedCalendarName,
                status, createdAt, updatedAt }

/users/{uid}/scanRuns/{timestamp}  ← scan run history
/users/{uid}/categories/{categoryId}
/users/{uid}/categoryHints/{hintId}
/users/{uid}/payments/{paymentId}
/users/{uid}/outbound/{outboundId}
/users/{uid}/meta/usage            ← aggregated cost tracking
/users/{uid}/meta/scanFeed         ← live scan progress (overwritten per scan)
/users/{uid}/meta/usage_archive_{timestamp}  ← archived before dev resets

/config/aiProvider  ← active AI provider string (read by aiComplete.ts)

Signal types: event | payment | rsvp | action | info | awaiting_reply | deadline

IMPORTANT — field naming:
  detectedAmountPence (not detectedAmount) — scan route, Cloud Function, reanalyse
  docToSignal reads both as fallback: d.detectedAmountPence ?? d.detectedAmount

═══════════════════════════════════════════════
KNOWN LIMITATIONS / WATCH POINTS
═══════════════════════════════════════════════

- Vercel serverless 5min timeout: onboarding uses Cloud Function automatically
  via NEXT_PUBLIC_SCAN_FUNCTION_URL. Incremental scans (~20s) fine on Vercel.
- Google OAuth app unverified: max 100 test users until verification submitted.
  Non-sensitive scopes (gmail.readonly, gmail.send, calendar) — standard review.
  Restricted scopes (gmail.modify) would require $15–75k CASA audit — avoid.
- Gemini free tier quota = 0 for some endpoints: billing must be enabled.
- Firestore eur3 region: data stays in EU (Frankfurt/Netherlands).
- No field-level encryption yet: PII in Firestore is at-rest encrypted by GCP only.
- Multiple local docToItem() copies: awaiting-reply/page.tsx, quietly-logged/page.tsx,
  Topbar.tsx, ItemExpandedPanel.tsx, hooks.ts all have local versions.
  Any new KeelItem field must be added to ALL copies — fragile, needs centralising.
- KEEL_APP_URL must be set in keel-admin Vercel env vars for Dev Operations to work.
- detectedAmountPence is the canonical field name for signal amounts. docToSignal
  reads both detectedAmountPence and detectedAmount as fallback for old data.
