Last updated: 2026-05-10 Session 6

═══════════════════════════════════════════════
OVERVIEW
═══════════════════════════════════════════════

Keel is a Gmail-connected inbox intelligence layer. It scans a user's Gmail,
classifies threads by category using AI, and surfaces actionable signals
(payments due, events, awaiting replies etc.) in a structured dashboard.

Two separate Next.js apps:
  keel       — user-facing app  → www.jaison.app / keel-icl6.vercel.app
  keel-admin — internal admin   → keel-admin.vercel.app

IMPORTANT: The active dashboard is DashboardShell2 at /dashboard2 (not /dashboard1).
dashboard2/page.tsx wraps DashboardShell2 in CategoryFilterProvider.

═══════════════════════════════════════════════
SCAN ARCHITECTURE
═══════════════════════════════════════════════

THREE SCAN PATHS:

1. MANUAL / AUTO (Vercel serverless, ~20s):
   Client → triggerScan('manual') → POST /api/gmail/scan
   → reads OAuth token from Firestore server-side
   → Gmail API: existence + recent query (incremental fetch)
   → classifyThread() from scanUtils.ts (S1)
   → write items, scanRun doc, meta/usage

2. ONBOARDING (Cloud Function, 60min timeout):
   Client → triggerScan('onboarding') → POST https://gmailscan-wel4wongwa-ew.a.run.app
   → same logic, handles full 7–30 day window without Vercel timeout

3. BACKGROUND (Pub/Sub triggered, ~3s per email):
   New email arrives in Gmail
   → Gmail pushes to Pub/Sub topic gmail-inbox-notifications
   → handleGmailNotification CF fires
   → checks autoScanEnabled + 90s debounce on lastBackgroundScanAt
   → calls POST /api/gmail/background-scan on Vercel
   → getChangedThreadIds() via Gmail History API
   → classifyThread() from scanUtils.ts (same as manual)
   → write items, scanRun (job='background'), meta/usage backgroundScan* fields

Watch renewal: renewGmailWatches CF runs every 6 days (watches expire at 7).
Nightly expiry: nightlyItemExpiry CF runs at 1am — past event items → quietly_logged/overdue.

SHARED CLASSIFICATION (src/lib/scanUtils.ts):
All three scan paths import classifyThread() from scanUtils.ts.
Improving the prompt improves all scan types simultaneously.
Also exports: decodeBody(), buildThreadContext(), runInBatches()
decodeBody() extracts structured HTML data (dates/times/locations) before tag-stripping —
critical for invitation emails (Paperless Post, Eventbrite etc.)

Live feed: scan route writes meta/scanFeed every 5 threads during onboarding.

ScanRun logging: every scan writes to users/{uid}/scanRuns/{timestamp}
Fields: scanAt, daysBack, threadsFound, threadsProcessed, newItems, updatedItems,
        skipped, inputTokens, outputTokens, aiCostUsd, fbReads, fbWrites,
        fbCostUsd, totalCostUsd, model, provider, job, durationMs
job values: 'onboarding' | 'manual' | 'auto' | 'background' | 'reclassify'

═══════════════════════════════════════════════
FIRESTORE DATA MODEL
═══════════════════════════════════════════════

/users/{uid}                        ← root doc: email, autoScanEnabled, watchStatus,
                                      watchProvider, watchExpiry, watchHistoryId,
                                      lastBackgroundScanAt
/users/{uid}/accounts/account_primary ← accessToken, refreshToken, tokenExpiresAt,
                                         email, displayName, daysBack, categoryIds[],
                                         categoryDescriptions{}, emailType,
                                         checkAllCalendars, lastScanAt, scanCount

/users/{uid}/items/{itemId}
  KeelItem: { itemId, threadId, messageId, accountId, senderEmail, senderName,
              subject, receivedAt, categoryId, categoryName, status,
              aiImportanceScore, manualPriority, manuallyIgnored, manualCategory,
              mergedThreadIds[], aiTitle, aiSummary, aiDetailedSummary,
              participants[], signals[], createdAt, updatedAt, resolvedAt,
              snoozedUntil, isRecurring, lastProcessedBy, expiredBy,
              lastMessageInternalDate, noteToSelf }

/users/{uid}/scanRuns/{timestamp}   ← per-scan cost/token logging
/users/{uid}/categories/{catId}     ← category definitions
/users/{uid}/categoryHints/{id}     ← user correction hints for AI
/users/{uid}/signals/{sigId}        ← extracted signals with calendarStatus
/users/{uid}/payments/{id}          ← payment records

/meta/usage                         ← aggregated cost tracking (all users)
  backgroundScanRuns, backgroundScanCostUsd, backgroundScanInputTokens, etc.
/meta/scanFeed                      ← live scan progress (overwritten per scan)
/meta/usage_archive_{timestamp}     ← archived before dev resets

/config/aiProvider                  ← active AI provider string

Item status values:
  new | awaiting_reply | awaiting_action | quietly_logged | archived | overdue

Signal types: event | deadline | payment | rsvp | awaiting | info
SignalCalendarStatus: 'on_cal' | 'not_on_cal' | 'ignored' | 'pending' | null

═══════════════════════════════════════════════
KNOWN LIMITATIONS / WATCH POINTS
═══════════════════════════════════════════════

- Vercel 5min timeout: onboarding uses Cloud Function. Incremental ~20s is fine.
- Google OAuth unverified: max 100 test users until verification submitted.
- Gemini billing must be enabled for production scale (free tier quota = 0).
- Background scan cost hardcoded as 'gemini-flash' in route — should read aiComplete.
- GCal API maxResults=2500 (hard limit) — paginate if users have very busy calendars.
- dashboard1/DashboardShell.tsx still exists but is unused — can be deleted.
- No field-level encryption: PII in Firestore encrypted at rest by GCP only (V2 roadmap).
